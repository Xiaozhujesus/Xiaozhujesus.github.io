Paxos

24 Apr 2020, 19:16

共识问题的要求

  • safety
  1. 只能选择已经被提出的值
  2. 最终只有一个提案值被选定，但是节点可以选择临时的提案值
  • liveness
  1. 所有正常节点可以在有限时间内对单个提案达成共识，并且系统可以持续对外提供服务

FLP 定理

异步系统中的一组进程无法对一个布尔类型的值达成共识，即使只有单个进程出现未知的故障，核心问题是异步

定理指出：在异步系统中，不可能保证 safety 和 liveness 两个性质同时满足，即使出现一点未知的故障；但是定理没有说明，我们可以有多接近同时满足这两个性质

备份状态机

通常使用日志备份实现，所有 server 以相同的顺序执行相同的命令，这样如果每个 server 的初始状态相同，那么最终状态也相同；日志备份是关键，需要保证每个 server 上的日志内容一致，使用共识算法对日志的第 i 条记录应该是什么命令达成共识

Paxos

completely-safe 并且 largely-live 的共识协议

Basic Paxos

  • 提案：例如，日志的第 i 条记录是什么命令
  • 提案值：例如，日志的第 i 条记录是 x=1，即将 x 的值赋值为 1

基本 paxos 是指对单提案达成共识，例如，日志第 10 条记录应该是 x=1

  • 一个或者多个节点可以针对相同的提案提出自己的提案值
  • 只有一个提案值可以被最终选定

Multi Paxos

对日志文件的不同记录，使用不同的 Basic Paxos 来确定相应记录的值

系统角色：

  • Proposers：处理 client 请求，提出提案，并根据 acceptor 的响应最终决定 commit 还是开启新一轮提案
  • Acceptors：响应 Proposers 提出的提案，accept/reject；本地记录决策过程；主动学习被选择的提案值

每个节点可以实现超过一种角色，实际实现中，某个节点通常同时实现 2 种角色

问题

关键的一点是，每个节点提出的提案值应该是可以改变的，否则，如果一开始就有两个节点的提案值不同，而每个节点都坚持自己最初的提案值而不进行更改，那么永远也无法达成共识，无论执行多少轮共识算法；因此，第一阶段应该是提出提案尝试进行询问，如果已经有选定的提案值，会在响应中得到提案值；如果没有，那么第二阶段可以尝试提出提案值，进行广播提交

从 2-PC 可以看出，单个协调者会导致系统失去 liveness 性质，因此，可以接受 client 请求并且提出提案或提案值的节点应该有多个；而每个节点都是对等的，因此每个节点都可以提出提案或提案值，因此，每个节点会收到多个节点提出的提案或提案值，这里我们只考虑针对同一个提案，多提案可以使用多轮单提案算法；

在 2-PC 中，单个参与者如果出现问题，系统也会失去 liveness 性质；因此，对一个提案值的选定不应该要求所有 acceptor 都有响应，而为了保证 safety 性质，至少需要有超过半数的节点返回响应才可以做决策（commit/新一轮 Paxos），这样恢复节点时候，才可以保证不会丢失决策信息

对于 acceptor 收到（receive）的多个提案值，不能只接受（accept）第一次收到的提案值，否则会导致分票（因为通常都是先收到自己的提案值和临近节点的提案值），无法选出提案值，最终系统失去 liveness；

注意：

  • Paxos 算法只有 proposer 知道最终选定了哪个提案值
  • 每个节点都需要通过执行 Paxos 算法来得到最终提案，进而所有节点针对某个提案的值达成共识

例如，共有 5 个节点，node-1 到 node-5，如果只有 node-1 和 node-5 提出提案，但是由于网络原因，node-4 每次都是什么信息都没有收到，那么正常情况下最终 1-3 和 5 号节点会达到一致状态，但是 4 是不知道的；这里 node-2，node-3 没有执行 Paxos 算法也得到了提案值，但是就像 node-4 一样，你不知道这个提案值是不是正确的，如果想要知道，需要主动执行 Paxos 算法

每个 proposer 提出的提案需要进行区分，如果是针对单一提案，我们可以使用节点标识符进行区分；当多提案时候，每个节点本身提出的多个提案也需要相互区分

Basic Paxos 的 2-PC

流程： 假如 n 是初次选出最终提案值的提案号，那么不存在 m，m > n，m 所在节点的提案值不是最终提案值 也就是一定不会存在提案号大于已经选定提案值的提案号，但是其提案值却不是最终提案值的情况；

证明： n 是初次选出最终提案值的提案号，令 m = n+1，现在假设 m 所在节点的提案值不是最终提案值；m 的提案一定要经过第一阶段询问阶段，现在提案已经选定，那么 m 提案的第一阶段询问，会得到提案 n 的提案值，并使用该提案值作为 m 第二阶段的提案值，递归可得，任何提案号大于 n 的提案，其提案值必定是提案 n 的提案值，也就是最终选定的提案的提案值，因此不存在 m，m > n，m 所在节点的提出的提案值不是最终提案值

选定最终提案一定是有超过半数节点接受了提案值，没有 reject，所以要想产生上面的情况，以 3 个节点为例，就需 2 个节点是已经选定的提案值，第三个节点的提案号 n3 大于另外两个节点的提案号 n1 和 n2（n1=n2），但是该节点的提案值却不是最终提案值 一个节点更新他的提案值必须经过第一阶段，询问超过半数节点，而且没有 reject，而 n3 的第一阶段一定是在 n1 的第二阶段之后到达 node-1 和 node-2 的(否则没办法选定 n1 和 n2)，因此 n3 的第一阶段一定会得到最终选定的提案值，因此任何节点如果提案号是 n3，那么提案值一定是最终提案值
