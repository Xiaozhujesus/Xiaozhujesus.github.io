Linearizability

14 Mar 2020, 07:56

动机

java 和 C++中的对象其实是数据容器，每个对象都定义了一组方法对对象的状态进行操作，对象的类描述了方法的行为，描述对象的行为有很多方法：例如 API；在 sequential model 中，单线程对一组对象操作，不会有什么问题，当多线程共享相同的对象时候，原来的模型失效了，所以需要新的模型来描述对象，这就是 Linearizability;

Linearizability 是并发对象（concurrent object）的正确性条件（correctness condition），也是 execution history 的一个属性，允许程序员使用 sequential model 中的已经存在的技术来说明（specify）和推理（reason about）并发对象；当多个进程并发调用并发对象的方法时候，Linearizability 假设，单个调用在 invocation 和对应的 response 之间的某个点瞬时生效，这个生效的瞬间叫 linearizatioon point；

下面是并发 FIFO 队列的实现：

class LockBasedQueue<T> {
    int head, tail;
    T[] items;
    Lock lock;

    public LockBasedQueue(int capacity) {
        head = 0;
        tail = 0;
        ock = new ReentrantLock();
        items = (T[]) new Object[capacity];
    }
    public T deq() throws EmptyException {
        lock.lock();
        try {
            if (tail == head)
                throw new EmptyException();
            T x = items[head % items.length];
            head++;
            return x;
        } finally {
            lock.unlock();
        }
    }
}

上面的实现应该是正确的，因为所有的修改操作都是互斥的，使用了锁；下面让我们考虑另一种实现，Wait-free 2-thread Queue

public class WaitFreeQueue {

    int head = 0, tail = 0;
    items = (T[]) new Object[capacity];

    public void enq(Item x) {
        if (tail-head == capacity)
            throw new FullException();
        items[tail % capacity] = x;
        tail++;
    }
    public Item deq() {
        if (tail == head)
            throw new EmptyException();
        Item item = items[head % capacity];
        head++;
        return item;
    }
}

这里没有使用锁，我们怎么知道实现是正确的呢？

定义并发队列实现

通常，为了使我们直观地了解算法是正确的，我们需要一种说明（specify）并发队列对象的方式；需要一种方式来证明算法确实实现了对象的规范（specification）

并发规范对实现施加两个元素，safety 和 liveness，本文中也叫 correctness 和 progress，这里主要讨论 correctness

sequential specification

使用前置条件和后置条件来说明（pre-condition，post-condition）

  • 前置条件，定义了调用方法之前对象的状态
  • 后置条件，定义了调用方法之后对象的状态，同时定义了返回值和异常

例如 deq 方法

前置条件：队列不空；后置条件：返回队列的第一个元素并从队列中删除

前置条件：队列空；后置条件：抛 EmptyException，队列状态不变

在 sequential specification 中，上面的方式没问题，你不需要考虑方法间的交互，而且可以很容易添加新的方法，不需要改变旧的方法

concurrent specification

这里我们需要理解，方法的调用执行是需要时间的，在 sequential 计算时，方法执行同样需要时间，但是我们不关心

  • sequential：方法调用是 event
  • concurrent：方法调用是 interval

不同线程的方法调用也可能有时间重叠

sequential concurrent
每个方法独立描述 需要描述方法间所有可能的交互（例如 enq 和 deq overlap 了）
对象的状态在方法调用之间定义 由于方法调用可能出现 overlap，对象有可能永远不会出现在方法调用之间
加新方法不影响旧的方法 加新的方法时，需要考虑所有可能与旧方法的交互

解决这种困境的方法就是 Linearizability

Linearizability

Linearizability 声明：

Each method call should appear to “take effect” instantaneously at some moment between its invocation and response.

此声明不是科学声明，不能被证明或证伪；但是，它作为一种正确性已经得到广泛接受

  • Linearizability 的第一个优势就是不需要考虑大量方法之间的交互
  • 因此我们可以使用 sequential 中的 pre-condition 和 post-condition
  • 即使如此，还是会有不确定性；例如，如果 x 和 y 被不同进程以 overlap interval 的方式调用了 enq 方法，那么之后的 deq 调用可能返回 x，也可能返回 y，但不可能是 z 或者抛出异常

Linearizability 有两个重要属性

local property

系统是 linearizable 的当且仅当系统中每个单独的对象是 linearizable 的；这给了我们可组合性（composability）

non-blocking property

方法之间不需要强制等待同步

形式模型

上面这种识别操作原子生效的方法（“linearization points”）是表明可线性化的最常用方法。linearization points 依赖于执行，我们需要一个形式模型来定义 linearizability 和其他 correctness conditions

method call 包括两个 event：

  • Invocation，包括 method name 和 args，例如：q.enq(x)
  • Response，包括 result 或 exception，例如：q.enq(x) returns void，q.deq() returns x or throws emptyException

符号说明：

Invocation：A q.enq(x)

  • A – thread
  • q – object
  • enq – method
  • x – arg

Response：A q: void , A q: empty()

  • A – thread
  • q – object
  • void – result, exception

History

描述了 execution，是由 invocation 和 response 组成的序列，例如下面是一个 history

  • A q.enq(3)
  • A q:void
  • A q.enq(5)
  • B p.enq(4)
  • B p:void
  • B q.deq()
  • B q:3

match

如果 thread names 和 object names 匹配，我们就说 Invocation 和 Response 匹配，例如

  • A q.enq(3)
  • A q:void

Object projection:

H|q =

  • A q.enq(3)
  • A q:void
  • A q.enq(5)
  • B q.deq()
  • B q:3

Thread projection:

H|A =

  • A q.enq(3)
  • A q:void
  • A q.enq(5)

pending invocation:

没有与之匹配的 response 的 invocation

  • A q.enq(3)
  • A q:void
  • A q.enq(5) —> pending invocation
  • B q.deq()
  • B q:3

Complete history:

没有 pending invocation 的 history

Sequential history

由 match 的 invocation 和 response 组成的序列，可以存在 pending invocation，例如

  • A q.enq(3)
  • A q:void
  • B p.enq(4)
  • B p:void
  • B q.deq()
  • B q:3
  • A q:enq(5)

Well-formed history

对于每个线程 A，H|A 都是 Sequential 的

Equivalent history

如果对于每个线程 A，H|A = G|A，那么 H 和 G 是等价的，例如

H =

  • A q.enq(3)
  • B p.enq(4)
  • B p:void
  • B q.deq()
  • A q:void
  • B q:3

G =

  • A q.enq(3)
  • A q:void
  • B p.enq(4)
  • B p:void
  • B q.deq()
  • B q:3

假设 H =

  • A q.enq(3)
  • B p.enq(4)
  • B p:void
  • B q.deq()
  • A q:void
  • B q:4

那么 H 和 G 不等价，H|A 和 G|A 有点类似从 client 角度看，而 H 和 G 类似是从 client 调用的系统内部看全局（系统小到可能是一个数据结构，也有可能是个系统比如 zk），如果每个 client 看到的效果都一样，那么不论系统内部总的 history 的 inv 和 resp 顺序如何排列，history 就是等价的

方法调用优先关系（precede）

如果 method1 的 response 在 method2 的 invocation 之前，那么方法调用 method1 优先于方法调用 method2；符号表示如下： m1 ->(H) m2，注意，这只定义了偏序关系 例如：

H =

  • A q.enq(3)
  • B p.enq(4)
  • B p.void
  • A q:void
  • B q.deq()
  • B q:3

method overlap

H =

  • A q.enq(3)
  • B p.enq(4)
  • B p.void
  • B q.deq()
  • A q:void
  • B q:3

Sequential Specifications

是一种判断单线程，单对象的 history 是否合法的方法；有许多种方法，我们见过一种，就是使用前置条件和后置条件；sequential specification 是每个对象的属性，不同对象的 sequential specification 不同，例如，对于 queue 而言，如果 q.enq(x)，则下个 q.deq()必定返回 x，如果返回其他值，则不满足 queue 的 sequential specification；对于 stack 而言，其 sequential specification 为，如果最后 s.push(x)，即最后 push 的是 x，那么紧接着的 s.pop()必定返回 x，如果返回其他值就不满足 stack 的 sequential specification

Legal history

如果对于每个 object x，H|x 满足 x 的 sequential specifications，sequential history H 是 legal 的，例如 object 可以是 queue, stack；换句话说，不论是单线程还是多线程，单看单个 object 的 history，由 object 本身决定的 sequential specification 必须满足，否则不合法

Linearizability 形式化定义

History H 是 linearizable 的，如果 H 可以扩展为 G，而 G 与 legal sequential history S 等价，G 上的偏序关系是 S 上偏序关系的子集，G 上的偏序关系表示为->(G)，同理 S 上的表示为->(S)；G 与 H 完全相同，只不过扩展之后消除了 pending invocations，可以使用如下两种方法进行扩展：

  • 给 pending invocations 添加 responses.
  • 丢弃 pending invocations

下面解释下，G 上的偏序关系是 S 上偏序关系的子集的含义，举个例子来说明：

->(G) = {a->c,b->c}

->(S) = {a->b,a->c,b->c}
