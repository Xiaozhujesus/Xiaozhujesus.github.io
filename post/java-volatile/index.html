Java Volatile

12 Apr 2020, 10:25

volatile 关键字标记 Java 变量在 main memory 而不是 cache，每次读取从 main memory 读，每次更新写回 main memory，除此之外还有一些其他的保证，下面介绍下

变量可见性问题

volatile 保证变量更新后的跨线程的可见性，看下面的例子：

public class SharedObject {
    public int counter = 0;
}

假设两个线程访问该变量，thread1 会更新变量，而 thread2 只读变量，如果变量 counter 不声明为 volatile，那么 thread1 对变量的更新不保证何时写回 main memory，更新可能只是存在 cache 中，所以此时 cache 和 main memory 中 counter 的值是不一致的；另外，即使 thread1 将新的值写回 main memory 了，由于 thread2 的所在 CPU 的 cache 中可能始终是旧值，所以也不一定能看到新的值

Java volatile 可见性保证

volatile 用于保证变量可见性问题，声明为 volatile 的变量，所有对变量的更新都会写回 main memory，而且对变量的访问也会从 main memory 读取，对于上面的情况，将变量声明为 volatile 足够了，但是如果两个线程都要更新变量，只是将变量声明为 volatile 还不够，还需要同步

完整的可见性保证

事实上，volatile 变量的可见性保证已经超出 volatile 变量本身，如下：

  • 如果 thread1 更新 volatile 变量，thread2 读这个变量，那么所有更新 volatile 变量之前对 thread1 可见的变量，在 thread2 读取 volatile 变量后对 thread2 都可见
  • 当 thread 读取 volatile 变量时候，所有对该线程可见的变量都会重新从 main memory 中读取

举个例子：

public class MyClass {
    private int years;
    private int months
    private volatile int days;

    public void update(int years, int months, int days){
        this.years  = years;
        this.months = months;
        this.days   = days;
    }
}

上面的 update 会更新 3 个变量，只有 day 是 volatile，完整可见性保证当 day 写回 main memory 时候，其余两个变量同样写回 main memory；当读取三个变量时候，可以这样操作

public class MyClass {
    private int years;
    private int months
    private volatile int days;

    public int totalDays() {
        int total = this.days;
        total += months * 30;
        total += years * 365;
        return total;
    }

    public void update(int years, int months, int days){
        this.years  = years;
        this.months = months;
        this.days   = days;
    }
}

注意，上面首先读取 day 变量，然后才是其余两个变量，当读取 day 时候，再读取其余两个变量也是从 main memory 读取，这样可以保证读取最新的值；但是如果读取 day 放在最后，那么前面读到的 year 和 month 可能是从 cache 中读的，导致读取到的不是最新的值，从而产生奇怪的问题

指令重排序

在保证语义不变的前提下，jvm 可以对指令进行重新排序，例如：

int a = 1;
int b = 2;

a++;
b++;

可以重排序为

int a = 1;
a++;

int b = 2;
b++;

当变量是 volatile 变量时候，重排序指令可能会改变语义；还是上面的例子，如果重排序后 update 方法变成下面这样

public void update(int years, int months, int days){
    this.days   = days;
    this.months = months;
    this.years  = years;
}

那么就会发生 month 和 year 的最新值可能对其他线程不可见，因为在 cache 中，没有刷回到 main memory；对于这种情况，Java 有解决方法，我们后面介绍

volatile 的 Happens-Before 保证

为了解决指令重排序的问题，volatile 不仅有可见性保证，还有 happen-before 保证：

  • 如果读或写其他变量原来是在更新 volatile 变量之前，那么指令重排序后，这些操作不可以在更新 volatile 变量之后；但是如果原来操作是在更新 volatile 变量之后，那么指令重排序后操作可以移动到更新 volatile 变量之前
  • 如果读或写其他变量原来是在读 volatile 变量之后，那么指令重排序后，这些操作不可以在读 volatile 变量之前；但是如果原来的操作是在读 volatile 变量之前，那么指令重排序后操作可以移动到读 volatile 变量之后

需要同步的情况

当多线程同时对 volatile 进行更新，但是更新的值不依赖于该变量之前的值的情况下，volatile 是够用的；但是如果更新操作依赖于该变量之前的值，换句话说，新的值需要基于旧值计算得到，则对该共享 volatile 变量的并发更新操作，volatile 不足以保证正确性；多个线程可能读到相同值，基于该值进行一些计算，然后写回 main memory，但是写回会发生覆盖现象；比如原来变量在 main memory 中是 0，两个线程同时读取该变量的值，然后进行加 1 操作，然后写回 main memory，此时变量值应该为 2，实际却为 1

上面这种情况需要使用 synchronized 或者 java.util.concurrent 中的原子类型变量或者锁一样的同步机制

性能问题

volatile 变量会影响指令重排序，而指令重排序是性能提升技术；另外，volatile 是操作 main memory，相比于操作 cache 来说，开销更大，性能更低；所以只有确定需要多线程环境下的更新操作，需要保证变量可见性时候，才使用 volatile
